# FSN-0026: SIWE Login Implementation Plan

## Overview

Add Sign-In with Ethereum (SIWE / EIP-4361) authentication to the Fusion dashboard, restricting access to whitelisted Ethereum addresses. Uses wagmi v3 hooks for wallet connection + signing, Supabase `signInWithWeb3()` (manual mode) for session management, and Next.js middleware for route protection.

## Current State Analysis

- **No auth exists** — no middleware, login pages, session management, or protected routes
- **wagmi 3.4.2 + viem 2.45.1** configured for 3 chains (mainnet, arbitrum, base) — read-only, no connectors
- **RainbowKit NOT installed** — and incompatible with wagmi v3, so we won't use it
- **Supabase** used for data only via service-role client (`@wgenie/fusion-supabase-ponder`)
- **AppProviders** (QueryClient + Wagmi) applied per-page, not at root layout
- **Local Supabase** running at `http://127.0.0.1:54341`

### Key Discoveries:

- `packages/web/src/app/wagmi-provider.tsx` — wagmi config with no connectors, only HTTP transports
- `packages/web/src/app/layout.tsx` — root layout has no providers, just `<html><body>{children}</body></html>`
- `packages/web/src/app/(dashboard)/layout.tsx` — wraps in `SidebarLayout` (client component)
- `packages/web/src/components/sidebar/sidebar-layout.tsx` — sidebar + header, no auth UI
- No `middleware.ts` exists
- Supabase has native `signInWithWeb3({ chain, message, signature })` — manual mode lets us handle wallet connection ourselves

## Desired End State

1. Unauthenticated users are redirected to `/login`
2. Login page shows a "Connect Wallet" button using wagmi v3's `injected()` connector
3. After wallet connects, the app checks the address against a `whitelisted_addresses` Supabase table
4. If not whitelisted: shows "Address not authorized" error
5. If whitelisted: creates EIP-4361 SIWE message, user signs it, calls `supabase.auth.signInWithWeb3()` with message + signature
6. Supabase creates a session (cookie-based via `@supabase/ssr`)
7. User is redirected to the dashboard
8. Middleware refreshes sessions and enforces auth on all routes except `/login`
9. Sidebar shows connected address + sign out button

### Verification:

- Visit any page while unauthenticated → redirected to `/login`
- Connect a non-whitelisted address → see "Address not authorized" error
- Connect a whitelisted address → sign SIWE message → land on dashboard
- Refresh page → session persists (cookie-based)
- Click sign out → redirected to `/login`

## What We're NOT Doing

- No RainbowKit (incompatible with wagmi v3)
- No WalletConnect / mobile wallet support (injected connector only for now)
- No RLS on existing data tables (they use service-role client, unchanged)
- No email/phone linking to Web3 accounts
- No CAPTCHA (internal tool with whitelisting is sufficient)
- No rate limiting configuration (can be added later)

## Implementation Approach

**Auth flow**: wagmi `useConnect` (injected) → whitelist check (Route Handler) → viem `createSiweMessage` → wagmi `useSignMessage` → Supabase `signInWithWeb3({ chain: 'ethereum', message, signature })` → cookie session via `@supabase/ssr` → middleware protection.

**Two Supabase clients**:
1. **Auth client** (anon key, `@supabase/ssr`) — for auth operations + session management
2. **Data client** (service-role, existing `@wgenie/fusion-supabase-ponder`) — unchanged, for data queries

import { createServerClient } from '@supabase/ssr';
import { cookies } from 'next/headers';

export async function createSupabaseServerClient() {
  const cookieStore = await cookies();

  return createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll() {
          return cookieStore.getAll();
        },
        setAll(cookiesToSet) {
          try {
            cookiesToSet.forEach(({ name, value, options }) =>
              cookieStore.set(name, value, options)
            );
          } catch {
            // Called from Server Component — middleware handles refresh
          }
        },
      },
    }
  );
}

import { NextResponse, type NextRequest } from 'next/server';
import { createSupabaseServerClient } from '@/lib/supabase/server';

export async function GET(request: NextRequest) {
  const address = request.nextUrl.searchParams.get('address');

  if (!address) {
    return NextResponse.json({ error: 'Address required' }, { status: 400 });
  }

  const supabase = await createSupabaseServerClient();
  const { data, error } = await supabase
    .from('whitelisted_addresses')
    .select('address')
    .eq('address', address.toLowerCase())
    .single();

  if (error || !data) {
    return NextResponse.json({ whitelisted: false });
  }

  return NextResponse.json({ whitelisted: true });
}

#### 4. Create login content component

**File**: `packages/web/src/auth/login-content.tsx` (NEW)

This is the main login UI. It handles:
1. Wallet connection via `useConnect` + `injected()`
2. Whitelist check via fetch to `/api/auth/check-whitelist`
3. SIWE message creation via `createSiweMessage` from `viem/siwe`
4. Message signing via `useSignMessage`
5. Supabase `signInWithWeb3()` call
6. Redirect to `/` on success

```typescript
'use client';

import { useState } from 'react';
import { useAccount, useConnect, useDisconnect, useSignMessage } from 'wagmi';
import { injected } from 'wagmi/connectors';
import { createSiweMessage, generateSiweNonce } from 'viem/siwe';
import { useRouter } from 'next/navigation';
import { createSupabaseBrowserClient } from '@/lib/supabase/client';

export function LoginContent() {
  const router = useRouter();
  const { address, isConnected, chainId } = useAccount();
  const { connect, isPending: isConnecting } = useConnect();
  const { disconnect } = useDisconnect();
  const { signMessageAsync } = useSignMessage();

  const [status, setStatus] = useState<'idle' | 'checking' | 'signing' | 'authenticating' | 'error'>('idle');
  const [error, setError] = useState<string | null>(null);

  const handleConnect = () => {
    setError(null);
    connect({ connector: injected() });
  };

  const handleSignIn = async () => {
    if (!address || !chainId) return;

    try {
      // Step 1: Check whitelist
      setStatus('checking');
      setError(null);

      const res = await fetch(`/api/auth/check-whitelist?address=${address.toLowerCase()}`);
      const { whitelisted } = await res.json();

      if (!whitelisted) {
        setStatus('error');
        setError('Address not authorized. Contact an administrator to request access.');
        return;
      }

      // Step 2: Create SIWE message
      setStatus('signing');

      const message = createSiweMessage({
        domain: window.location.host,
        address,
        statement: 'Sign in to Fusion by wGenie',
        uri: window.location.origin,
        version: '1',
        chainId,
        nonce: generateSiweNonce(),
      });

      // Step 3: Sign message
      const signature = await signMessageAsync({ message });

      // Step 4: Authenticate with Supabase
      setStatus('authenticating');

      const supabase = createSupabaseBrowserClient();
      const { error: authError } = await supabase.auth.signInWithWeb3({
        chain: 'ethereum',
        message,
        signature,
      });

      if (authError) {
        throw new Error(authError.message);
      }

      // Step 5: Redirect to dashboard
      router.push('/');
      router.refresh();
    } catch (err) {
      setStatus('error');
      setError(err instanceof Error ? err.message : 'Sign-in failed. Please try again.');
    }
  };

  return (
    <div className="flex min-h-screen items-center justify-center bg-background">
      <div className="w-full max-w-sm space-y-6 rounded-lg border border-border bg-card p-8">
        <div className="flex flex-col items-center gap-4">
          <img
            src="/assets/logo-fusion-by-wGenie.svg"
            alt="Fusion by wGenie"
            className="h-12 w-auto"
          />
          <h1 className="text-xl font-semibold text-foreground">Sign In</h1>
          <p className="text-sm text-muted-foreground text-center">
            Connect your Ethereum wallet to access the dashboard.
          </p>
        </div>

        {error && (
          <div className="rounded-md bg-destructive/10 p-3 text-sm text-destructive">
            {error}
          </div>
        )}

        {!isConnected ? (
          <button
            onClick={handleConnect}
            disabled={isConnecting}
            className="w-full rounded-md bg-primary px-4 py-2.5 text-sm font-medium text-primary-foreground hover:bg-primary/90 disabled:opacity-50"
          >
            {isConnecting ? 'Connecting...' : 'Connect Wallet'}
          </button>
        ) : (
          <div className="space-y-4">
            <div className="rounded-md bg-muted p-3">
              <p className="text-xs text-muted-foreground">Connected as</p>
              <p className="font-mono text-sm text-foreground truncate">{address}</p>
            </div>

            <button
              onClick={handleSignIn}
              disabled={status === 'checking' || status === 'signing' || status === 'authenticating'}
              className="w-full rounded-md bg-primary px-4 py-2.5 text-sm font-medium text-primary-foreground hover:bg-primary/90 disabled:opacity-50"
            >
              {status === 'checking' && 'Checking access...'}
              {status === 'signing' && 'Sign the message in your wallet...'}
              {status === 'authenticating' && 'Signing in...'}
              {(status === 'idle' || status === 'error') && 'Sign In with Ethereum'}
            </button>

            <button
              onClick={() => {
                disconnect();
                setError(null);
                setStatus('idle');
              }}
              className="w-full rounded-md border border-border px-4 py-2 text-sm text-muted-foreground hover:bg-muted"
            >
              Disconnect Wallet
            </button>
          </div>
        )}
      </div>
    </div>
  );
}
```

## Phase 3: Middleware + Route Protection

### Overview

Create Next.js middleware to protect all routes except `/login`. Refresh Supabase sessions on every request.

### Changes Required:

#### 1. Create session update utility

**File**: `packages/web/src/lib/supabase/middleware.ts` (NEW)

```typescript
import { createServerClient } from '@supabase/ssr';
import { NextResponse, type NextRequest } from 'next/server';

export async function updateSession(request: NextRequest) {
  let supabaseResponse = NextResponse.next({ request });

  const supabase = createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll() {
          return request.cookies.getAll();
        },
        setAll(cookiesToSet) {
          cookiesToSet.forEach(({ name, value }) =>
            request.cookies.set(name, value)
          );
          supabaseResponse = NextResponse.next({ request });
          cookiesToSet.forEach(({ name, value, options }) =>
            supabaseResponse.cookies.set(name, value, options)
          );
        },
      },
    }
  );

  // Validate and refresh session
  const {
    data: { user },
  } = await supabase.auth.getUser();

  // Redirect unauthenticated users to /login
  if (
    !user &&
    !request.nextUrl.pathname.startsWith('/login') &&
    !request.nextUrl.pathname.startsWith('/api/auth')
  ) {
    const url = request.nextUrl.clone();
    url.pathname = '/login';
    return NextResponse.redirect(url);
  }

  // Redirect authenticated users away from /login
  if (user && request.nextUrl.pathname.startsWith('/login')) {
    const url = request.nextUrl.clone();
    url.pathname = '/';
    return NextResponse.redirect(url);
  }

  return supabaseResponse;
}
```

Note: Using `getUser()` instead of `getClaims()` — `getUser()` always validates with the Supabase Auth server and is the recommended approach for middleware. If `getClaims()` is available in the installed version, we can switch to it for better performance.

**File**: `packages/web/src/auth/use-auth.ts` (NEW)

```typescript
'use client';

import { useCallback, useEffect, useState } from 'react';
import { useRouter } from 'next/navigation';
import { createSupabaseBrowserClient } from '@/lib/supabase/client';
import type { User } from '@supabase/supabase-js';

export function useAuth() {
  const router = useRouter();
  const [user, setUser] = useState<User | null>(null);
  const supabase = createSupabaseBrowserClient();

  useEffect(() => {
    supabase.auth.getUser().then(({ data: { user } }) => {
      setUser(user);
    });

    const {
      data: { subscription },
    } = supabase.auth.onAuthStateChange((_event, session) => {
      setUser(session?.user ?? null);
    });

    return () => subscription.unsubscribe();
  }, [supabase]);

  const signOut = useCallback(async () => {
    await supabase.auth.signOut();
    router.push('/login');
    router.refresh();
  }, [supabase, router]);

  // Extract Ethereum address from user identity
  const address = user?.identities?.[0]?.identity_data?.address as string | undefined;

  return { user, address, signOut };
}
```

## Implementation Notes (Discovered During Development)

- **Supabase anon key**: Local Supabase uses ES256 JWT keys (not the old HS256 demo keys)
- **Redirect URLs**: SIWE message URI must match Supabase's `site_url` / `additional_redirect_urls` — include all localhost variants
- **Address casing**: Whitelist stores lowercase, added DB trigger to auto-lowercase on insert
- **User metadata path**: Supabase Web3 stores the address at `user.user_metadata.custom_claims.address` (not `user_metadata.address`)
- **Sidebar providers**: `SidebarUser` wraps itself in `AppProviders` since the sidebar sits outside the per-page wagmi/react-query providers

## Performance Considerations

- Middleware runs on every request — `getUser()` call adds latency. Can switch to `getClaims()` if available for JWT-only validation (no network call).
- Browser Supabase client is created per-component — consider memoizing if needed.
- No impact on existing data queries (they use the separate service-role client).

## Migration Notes

- **Existing Supabase data client** (`@wgenie/fusion-supabase-ponder`) is unchanged — continues using service-role key for server-side data queries
- **New auth client** uses anon key + `@supabase/ssr` for cookie-based auth
- The two clients are completely independent — different keys, different purposes
- When deploying to production, need to:
  1. Enable Web3 Ethereum provider in Supabase dashboard
  2. Set `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY` env vars
  3. Configure Redirect URLs in Supabase to match production domain
  4. Populate `whitelisted_addresses` table with real addresses

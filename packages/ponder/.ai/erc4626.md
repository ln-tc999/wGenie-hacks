# ERC-4626: Tokenized Vaults - Events and Methods Reference

## Overview

ERC-4626 is an Ethereum standard for **Tokenized Vaults** that represent shares of a single underlying EIP-20 token. This standard provides a unified interface for yield-bearing vaults, making integration easier across DeFi protocols.

### Key Concepts

- **Asset**: The underlying EIP-20 token managed by the vault
- **Share**: The vault token representing a claim to underlying assets
- **Vault**: Smart contract implementing ERC-4626 that manages assets and issues shares

## Events Emitted by ERC-4626 Vaults

ERC-4626 defines **two primary events** that track all deposit and withdrawal activities:

### 1. Deposit Event

```solidity
event Deposit(
    address indexed sender,
    address indexed owner,
    uint256 assets,
    uint256 shares
)
```

**Parameters:**

- `sender` (indexed): Address that initiated the deposit transaction
- `owner` (indexed): Address that received the vault shares
- `assets`: Amount of underlying tokens deposited
- `shares`: Amount of vault shares minted

**Emitted by methods:**

- `deposit(uint256 assets, address receiver)`
- `mint(uint256 shares, address receiver)`

### 2. Withdraw Event

```solidity
event Withdraw(
    address indexed sender,
    address indexed receiver,
    address indexed owner,
    uint256 assets,
    uint256 shares
)
```

**Parameters:**

- `sender` (indexed): Address that initiated the withdrawal transaction
- `receiver` (indexed): Address that received the underlying assets
- `owner` (indexed): Address that owned the vault shares being burned
- `assets`: Amount of underlying tokens withdrawn
- `shares`: Amount of vault shares burned

**Emitted by methods:**

- `withdraw(uint256 assets, address receiver, address owner)`
- `redeem(uint256 shares, address receiver, address owner)`

## Method-to-Event Mapping

### Deposit Operations → Deposit Event

| Method                      | Description                                    | Event Emitted |
| --------------------------- | ---------------------------------------------- | ------------- |
| `deposit(assets, receiver)` | Deposit exact amount of assets, receive shares | `Deposit`     |
| `mint(shares, receiver)`    | Mint exact amount of shares, pay with assets   | `Deposit`     |

**Both methods result in:**

- Assets transferred from caller to vault
- Shares minted to receiver
- `Deposit` event emitted

### Withdrawal Operations → Withdraw Event

| Method                              | Description                                  | Event Emitted |
| ----------------------------------- | -------------------------------------------- | ------------- |
| `withdraw(assets, receiver, owner)` | Withdraw exact amount of assets, burn shares | `Withdraw`    |
| `redeem(shares, receiver, owner)`   | Burn exact amount of shares, receive assets  | `Withdraw`    |

**Both methods result in:**

- Shares burned from owner
- Assets transferred from vault to receiver
- `Withdraw` event emitted

## Event Tracking Patterns

### For Analytics & Indexing

```typescript
// Track all vault activity
interface VaultActivity {
  // Deposit events
  deposits: {
    sender: string;
    owner: string;
    assets: bigint;
    shares: bigint;
    timestamp: number;
    method: 'deposit' | 'mint';
  }[];

  // Withdraw events
  withdrawals: {
    sender: string;
    receiver: string;
    owner: string;
    assets: bigint;
    shares: bigint;
    timestamp: number;
    method: 'withdraw' | 'redeem';
  }[];
}
```

### Event Filtering Examples

```solidity
// Filter deposits by specific user
event.topics[0] = keccak256("Deposit(address,address,uint256,uint256)")
event.topics[2] = userAddress // owner parameter

// Filter withdrawals by specific receiver
event.topics[0] = keccak256("Withdraw(address,address,address,uint256,uint256)")
event.topics[2] = receiverAddress // receiver parameter
```

## Important Notes for Developers

### Event Reliability

- **MUST be emitted**: All compliant ERC-4626 vaults must emit these events
- **Comprehensive tracking**: These two events capture all vault share movements
- **No additional events**: Standard doesn't define other mandatory events

### Integration Considerations

- Use events to track user positions and vault activity
- Events provide exact asset/share exchange rates at transaction time
- Combine with view functions for real-time vault state
- Consider EIP-20 Transfer events for share transfers between users

### Gas Considerations

- Events are much cheaper to emit than storing state
- Off-chain indexing via events is gas-efficient way to track history
- Use events for analytics, view functions for current state

## Common Use Cases

1. **Portfolio Tracking**: Monitor user deposits/withdrawals across multiple vaults
2. **Yield Calculation**: Track assets deposited vs. assets withdrawn over time
3. **Volume Analytics**: Aggregate deposit/withdrawal volumes by time period
4. **Share Price History**: Calculate historical share prices from event data
5. **User Activity**: Track all vault interactions for specific addresses

## Security Considerations

- **Event manipulation**: Events reflect on-chain state changes, but preview functions can be manipulated
- **Rounding**: Events show actual amounts after rounding, which may differ from preview estimates
- **Fee inclusion**: Event amounts include any fees charged by the vault
- **Failed transactions**: Only successful transactions emit events

This reference provides the foundation for building robust analytics and integration systems with ERC-4626 vaults.
